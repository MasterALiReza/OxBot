Single-file per-bot dispatcher pattern: each bot is a self-contained directory under `vpnbot/<BotName>/` (e.g. `Default`, `update`) containing its own `index.php` (main update loop), `config.php` (per-bot `$ApiToken`), `botapi.php` (Telegram API wrappers + global `$update` parsing into `$from_id`/`$text`/`$datain`...), `func.php` (shared helpers like `DirectPaymentbot`, `channel_check`), `keyboard.php`, `admin.php`, `text.json`, and a `version` file. The root `vpnbot/index.php` is empty — deployment points directly at a bot subdirectory's `index.php`. Each bot's `index.php` bootstraps by reading `version`, setting timezone (`Asia/Tehran`), requiring shared parent files from the repo root (`function.php`, `config.php`, `jdf.php`, `panels.php`, vendor autoload) plus its local modules, then dispatches on `$text` and `$datain` through a long `if/elseif` chain implementing a state machine driven by the `user.step` column in the `botsaz` database. External integrations are isolated behind two boundaries: `ManagePanel` (a class loaded via Composer autoload for Marzban/WGDashboard/ibsng panel CRUD) and the `telegram()` / `sendmessage()` / `Editmessagetext()` family in `botapi.php`. Per-user transient state is persisted both in the DB (`Processing_value*` columns) and in JSON files under `data/{from_id}/{from_id}.json` (wallet balance). Admin logic is gated by checking `$from_id` against `$admin_ids` (per-bot) and `$admin_idsmain` (global) before entering `admin.php` handlers.
